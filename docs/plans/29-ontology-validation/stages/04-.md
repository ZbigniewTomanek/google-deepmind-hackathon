3. **Ingest Document 8** (Adversarial — technical jargon)

   Tests whether debugging concepts like "EmbeddingBatchProcessor", "OOM killer",
   "memory watchdog" get proper types (Tool, Concept) rather than overly specific
   types like `EmbeddingBatchProcessor`, `OomKiller`, `MemoryWatchdog`.

   ```bash
   curl -s -X POST http://localhost:8001/ingest/text \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer claude-code-work" \
     -d '{
       "text": "Debugging the Memory Leak — Postmortem\n\nService: ingestion-worker (Python 3.12, FastAPI + asyncpg)\nIncident: OOM kill at 2 AM on March 20, pod restarted 3 times before alert fired.\n\nRoot cause: The EmbeddingBatchProcessor was accumulating numpy arrays in a list without clearing between batches. Each batch added ~50MB. After processing 200 episodes overnight, memory hit 12GB and triggered the OOM killer.\n\nFix: Added explicit del + gc.collect() after each batch. Also reduced batch size from 100 to 25 episodes. Memory now stable at ~800MB under sustained load.\n\nInvestigation tools used:\n- py-spy for CPU profiling (attached to running container)\n- memray for memory profiling (ran locally with replay data)\n- kubectl top pods for real-time memory monitoring\n- Grafana dashboard (grafana.internal/d/api-latency) for correlation with latency\n\nLessons:\n- Python'\''s garbage collector doesn'\''t handle numpy array cycles well\n- asyncpg connection objects hold references to large result sets\n- The pod memory limit (4GB) was too low for batch processing — increased to 8GB\n- Need memory profiling in CI (tracked as PLAT-1275)\n\nPrevention: Added a memory watchdog that logs warning at 70% limit and kills gracefully at 85%.        "metadata": {"source": "plan29-test", "doc": "adversarial-postmortem"}
   ```

4. **Wait for all extraction jobs to complete**

5. **Check for instance-level types** — this is the critical check
   ```bash
   # Check all schemas for instance-level types
   docker compose exec postgres psql -U neocortex -d neocortex -c \
     "SELECT schema_name FROM graph_registry ORDER BY schema_name;"
   ```

   For each schema, run:
   ```bash
   docker compose exec postgres psql -U neocortex -d neocortex -c \
     "SELECT name FROM {schema}.node_type
      WHERE name ~ '^[A-Z][a-z]+[A-Z]'
      ORDER BY name;"
   ```

   **Red flags** (should NOT exist):
   - `ActivityTrackerPro`, `AssetManagerPro`, `DreamJournal`, `LocationScout`, `InsightTimer`
   - `EmbeddingBatchProcessor`, `OomKiller`, `MemoryWatchdog`
   - `MacMiniM2`, `ProxmoxVe`, `PiHole`, `HomeAssistant`

   **Acceptable types** (reuse of existing):
   - `Tool` (for apps and debugging tools)
   - `Asset` or `Device` (for hardware)
   - `Concept` (for architectural concepts)
   - `Component` (for software components)
   - `Infrastructure` (for server setup)

6. **Check for type proliferation** — after 8 documents, type count should still be modest
   ```bash
   # Count across all schemas
   for schema in $(docker compose exec postgres psql -U neocortex -d neocortex -t -c \
     "SELECT schema_name FROM graph_registry;"); do
     echo "=== $schema ==="
     docker compose exec postgres psql -U neocortex -d neocortex -c \
       "SELECT COUNT(DISTINCT nt.id) AS active_node_types
        FROM $schema.node_type nt
        JOIN $schema.node n ON n.type_id = nt.id;"
   done
   ```

7. **Review ontology agent logs for adversarial docs** — check if the agent's
   `find_similar_types` tool prevented creating duplicate types
   ```bash
   grep "ontology_agent_complete" log/agent_actions.log
   ```
