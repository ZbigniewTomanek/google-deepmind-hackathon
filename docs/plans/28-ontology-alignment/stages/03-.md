# Stage 3: Agentic Ontology Agent with Tool Access

**Goal**: Replace the 0-shot structured-output ontology agent with a tool-using agent that explores the current ontology, validates its own proposals inline, and only creates types for genuine gaps. Most episodes should result in zero new types.
**Dependencies**: Stage 1 (validation heuristics must exist for inline `propose_type` checks), Stage 2 (domain seeds provide recommended types)

---

## Design Rationale

The current ontology agent sees a flat list of type names, produces new types in a
single shot, and the pipeline trusts whatever it returns. This causes hallucinated
types, instance-level types, and massive proliferation (see `resources/redesign_handoff.md`
for full analysis).

The fix: give the agent tools to **explore** the ontology (usage counts, example
entities, similarity search) and **validate** its own proposals before committing.
This turns "propose new types" (generative) into "classify into existing types;
identify genuine gaps" (selective) — a much easier task for weaker models.

**Key design constraints:**
- Downstream interface unchanged: after the ontology step, `node_types` and `edge_types`
  lists are updated and passed to the extractor. The pipeline still persists via
  `repo.get_or_create_node_type/edge_type`.
- Final output is still `OntologyProposal` (structured output). Tools are for
  exploration and validation; the final commit is structured.
- Tools query the graph through `MemoryRepository` (same pattern as librarian).
- `pydantic_ai.models.test.TestModel` must still work for unit tests.

**Defense-in-depth (three layers):**
The `propose_type` tool is NOT the only validation gate. Even if the agent skips it
or ignores its feedback, the system is still safe:
1. **`propose_type` tool** — inline feedback with retry (agent can self-correct)
2. **Hard cap** (Step 8) — limits type count per episode
3. **`get_or_create_node_type` in adapter** (`adapter.py:466-470`) — already calls
   `normalize_node_type()` at persistence time and returns `None` on `ValueError`.
   The pipeline handles `None` by logging a warning and skipping (`pipeline.py:137`).
   Stage 1's heuristics will be enforced here regardless of agent behavior.

  ```python
  ) -> list[tuple[TypeInfo, int, list[str]]]:
      """Find types with names similar to a query string.

      Returns list of (TypeInfo, usage_count, example_entity_names) tuples,
      sorted by similarity score descending.
      - usage_count: number of nodes/edges using this type
      - example_entity_names: up to 3 entity names of this type
      """
      ...
  ```

- File: `src/neocortex/db/adapter.py` — Implement using trigram similarity:
  ```sql
  SELECT t.id, t.name, t.description,
         similarity(t.name, $2) AS sim,
         COUNT(n.id) AS usage_count
  FROM {schema}.node_type t
  LEFT JOIN {schema}.node n ON n.type_id = t.id AND NOT n.forgotten
  WHERE similarity(t.name, $2) > 0.2
     OR t.name ILIKE '%' || $2 || '%'
     OR t.description ILIKE '%' || $2 || '%'
  GROUP BY t.id
  ORDER BY sim DESC
  LIMIT $3
  ```
  Then fetch up to 3 example entity names per matched type.

#### Tool: `find_similar_types`

```python
@agent.tool
async def find_similar_types(
    ctx: RunContext[OntologyAgentDeps],
    query: str,
    kind: str = "node",  # "node" or "edge"
) -> list[dict]:
    """Search existing types by name similarity.
    Use this to check if a type like the one you're considering already exists.

    Args:
        query: Type name to search for (e.g. "HealthCondition", "CAUSES")
        kind: "node" for node types, "edge" for edge types

    Returns:
        List of {name, description, usage_count, example_entities} dicts,
        sorted by similarity. Empty list if no repo is available.
    """
    if not ctx.deps.repo:
        return []
    results = await ctx.deps.repo.find_similar_types(
        ctx.deps.agent_id, query, kind=kind,
        target_schema=ctx.deps.target_schema,
    )
    return [
        {
            "name": t.name,
            "description": t.description or "",
            "usage_count": count,
            "example_entities": examples,
        }
        for t, count, examples in results
    ]
```

#### Tool: `get_ontology_summary`

```python
@agent.tool
async def get_ontology_overview(
    ctx: RunContext[OntologyAgentDeps],
    include_unused: bool = False,
) -> dict:
    """Get a snapshot of the current ontology with usage statistics.
    Call this once at the start to understand the type landscape before
    making any proposals.

    Args:
        include_unused: If False (default), only return types with usage_count > 0.
            Set to True to see all types including unused ones.

    Returns:
        Dict with node_types, edge_types (each with name, description,
        usage_count), total_nodes, total_edges.
    """
    if not ctx.deps.repo:
        return {"node_types": [], "edge_types": [], "total_nodes": 0, "total_edges": 0}
    summary = await ctx.deps.repo.get_ontology_summary(
        ctx.deps.agent_id, target_schema=ctx.deps.target_schema,
    )
    if not include_unused:
        summary["node_types"] = [t for t in summary["node_types"] if t["usage_count"] > 0]
        summary["edge_types"] = [t for t in summary["edge_types"] if t["usage_count"] > 0]
    return summary
```

Note: filtering unused types by default keeps the response manageable (~60 types
instead of ~230). The agent can still use `find_similar_types` to search across
ALL types (including unused) for specific queries.

#### Tool: `propose_type`

This is the key tool — it runs Stage 1 validation inline and checks trigram
similarity, giving the agent immediate feedback.

```python
@agent.tool
async def propose_type(
    ctx: RunContext[OntologyAgentDeps],
    name: str,
    description: str,
    kind: str = "node",  # "node" or "edge"
) -> dict:
    """Propose a new type with inline validation.
    Runs normalization checks and similarity search before accepting.
    Does NOT persist the type — just validates and returns feedback.

    Args:
        name: Proposed type name (PascalCase for node, SCREAMING_SNAKE for edge)
        description: What this type represents
        kind: "node" or "edge"

    Returns:
        {accepted: bool, normalized_name: str, reason: str, similar_existing: list}
        If rejected, reason explains why. If accepted but similar types exist,
        similar_existing lists them so you can reconsider.
    """
    from neocortex.normalization import normalize_node_type, normalize_edge_type

    # 1. Run Stage 1 normalization/validation
    try:
        if kind == "edge":
            normalized = normalize_edge_type(name)
        else:
            normalized = normalize_node_type(name)
    except ValueError as e:
        return {"accepted": False, "normalized_name": name, "reason": str(e), "similar_existing": []}

    # 2. Check for similar existing types
    similar: list[str] = []
    if ctx.deps.repo:
        results = await ctx.deps.repo.find_similar_types(
            ctx.deps.agent_id, normalized, kind=kind,
            limit=3, target_schema=ctx.deps.target_schema,
        )
        similar = [t.name for t, _count, _ex in results]

    # 3. Check if type already exists (exact match)
    existing_types = (
        ctx.deps.existing_node_types if kind == "node"
        else ctx.deps.existing_edge_types
    )
    if normalized in existing_types:
        return {
            "accepted": False,
            "normalized_name": normalized,
            "reason": f"Type '{normalized}' already exists. Reuse it.",
            "similar_existing": similar,
        }

    return {
        "accepted": True,
        "normalized_name": normalized,
        "reason": "Validation passed.",
        "similar_existing": similar,
    }
```

### 6. Simplify context injection

- File: `src/neocortex/extraction/agents.py`, the `@agent.instructions` function
  below `build_ontology_agent()`
- Simplify `inject_context()`. Since the agent now uses tools to explore the ontology,
  it no longer needs the full type list + descriptions + examples in the context
  injection. Reduce to:

```python
@agent.instructions
async def inject_context(ctx: RunContext[OntologyAgentDeps]) -> str:
    parts: list[str] = []
    if ctx.deps.domain_hint:
        parts.extend([
            f"Domain context: {ctx.deps.domain_hint}",
            "Propose types semantically appropriate for this domain.",
            "",
        ])
    if ctx.deps.recommended_node_types or ctx.deps.recommended_edge_types:
        parts.append("Recommended types for this domain (prefer these over inventing new):")
        if ctx.deps.recommended_node_types:
            parts.append("  Node types: " + ", ".join(ctx.deps.recommended_node_types.keys()))
        if ctx.deps.recommended_edge_types:
            parts.append("  Edge types: " + ", ".join(ctx.deps.recommended_edge_types.keys()))
        parts.append("")
    parts.extend([
        "Text to analyze:",
        ctx.deps.episode_text,
        "",
        f"Current ontology has {len(ctx.deps.existing_node_types)} node types "
        f"and {len(ctx.deps.existing_edge_types)} edge types.",
        "Use get_ontology_overview and find_similar_types to explore them.",
    ])
    return "\n".join(parts)
```

The full type list with descriptions and examples is no longer dumped into the
context. The agent retrieves what it needs via tools. This keeps the initial prompt
smaller and lets the agent focus on the episode text.
