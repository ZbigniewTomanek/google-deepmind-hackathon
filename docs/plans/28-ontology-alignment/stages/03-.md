# Stage 3: Ontology Agent Prompt Overhaul

**Goal**: Rewrite the ontology agent's system prompt and context injection to produce domain-appropriate types with stronger constraints, negative examples, and a type budget.
**Dependencies**: Stage 2 (domain seeds must exist to inject as recommendations)

---

## Steps

### 1. Extend the ontology agent system prompt

- File: `src/neocortex/extraction/agents.py`, function `build_ontology_agent()`
- Details: Extend and strengthen the existing multi-part system prompt tuple. Preserve
  the current naming convention guidance (PascalCase for nodes, SCREAMING_SNAKE for
  edges) and the existing reuse instructions. Add the following new sections to the
  tuple:

  **a) Stronger reuse mandate:**
  ```
  MANDATORY: Before proposing ANY new type, verify that no existing type covers
  the concept. If an existing type is 80% suitable, USE IT -- do not create a
  more specific variant. The cost of a new type is fragmentation; the cost of
  reuse is minor imprecision. Always prefer reuse.
  ```

  **b) Explicit type budget:**
  ```
  Budget: Propose at most 2 new node types and 2 new edge types per episode.
  Most episodes need ZERO new types -- the existing ontology should cover them.
  If you need more than 2, the existing ontology is likely sufficient and you
  should reuse more aggressively.
  ```

  **c) Negative examples (what NOT to do):**
  ```
  NEVER create instance-level types. A type must be reusable across many entities.
  BAD: "DishGreg", "DreamAiPresentation", "LocationSalCapeVerde"
  GOOD: "Dish", "Dream", "Location"

  NEVER create types that embed function call syntax or API references.
  BAD: "ActivityfunctiondefaultApi", "MentalstatecalldefaultApi"

  NEVER create overly specific compound types when a general type exists.
  BAD: "AssetSnowboardguards" when "Asset" exists
  BAD: "ArchitecturePatternProgrammingModel" when "ArchitecturePattern" exists
  ```

  **d) Domain-appropriate type guidance:**
  ```
  For personal knowledge (health, relationships, routines, dreams):
  - Use types like Person, Location, HealthState, Symptom, Emotion, Routine, Dream
  - Use relations like HAS_STATUS, EXPERIENCES_SYMPTOM, VISITS_LOCATION, HAS_PARTNER
  - Do NOT use technical/business types like Account, BusinessActivityCode, Bot, Pipeline

  For technical knowledge:
  - Use types like Tool, Component, Protocol, Schema, Algorithm, Model
  - Use relations like USES, IMPLEMENTS, DEPLOYED_ON, DERIVES_FROM
  ```

### 2. Inject domain seed types as recommendations

- File: `src/neocortex/extraction/agents.py`, function `inject_context()` (the
  `@agent.instructions`-decorated function below `build_ontology_agent()`)
- Details: Read `recommended_node_types` and `recommended_edge_types` from
  `ctx.deps` (added to `OntologyAgentDeps` in Stage 2 Step 3). If non-empty,
  add a new section to the context injection that shows the domain's recommended
  types. Format:

  ```
  Recommended types for this domain:
  Node types: Person, Location, HealthState, Symptom, ...
  Edge types: HAS_STATUS, EXPERIENCES_SYMPTOM, VISITS_LOCATION, ...

  Prefer these over inventing new types. Only propose a new type if none of the
  recommended or existing types cover the concept.
  ```

  This is injected BEFORE the existing types list, so the model sees recommendations
  first, then the current state.

### 3. Add type count enforcement in the pipeline

- File: `src/neocortex/extraction/pipeline.py`, after the ontology agent `await` call
  and before the "persist new types" block
- Details: If the ontology agent proposes more than `MAX_NEW_TYPES_PER_EPISODE` (default 3)
  node or edge types, log a warning and truncate to the first N by order. This is a
  hard safety valve -- the prompt's soft budget of 2 should keep most episodes within
  bounds, but the hard limit catches outliers.

  ```python
  MAX_NEW_TYPES = 3
  if len(ontology_result.output.new_node_types) > MAX_NEW_TYPES:
      logger.warning("ontology_type_budget_exceeded",
                     proposed=len(ontology_result.output.new_node_types),
                     limit=MAX_NEW_TYPES)
      ontology_result.output.new_node_types = ontology_result.output.new_node_types[:MAX_NEW_TYPES]
  # Same for edge types
  ```

  Make `MAX_NEW_TYPES` configurable via `MCPSettings.ontology_max_new_types` (default 3).

### 4. Update tests

- File: `tests/test_extraction*.py`
- Details: Adjust any extraction tests that assert on the ontology agent's output format
  to account for the new prompt structure. Ensure test models still produce valid output.
