# Stage 2: Domain-Specific Seed Ontologies

**Goal**: Expand the 6-type seed ontology into rich, domain-specific type sets so the ontology agent starts from a meaningful foundation instead of inventing everything from scratch.
**Dependencies**: Stage 1 (validation must be in place before new types are seeded)

---

## Steps

### 1. Expand the base graph seed ontology

- File: `migrations/graph/003_seed_ontology.sql`
- Details: Expand from 6 node types + 12 edge types to a richer base that covers
  universal concepts. These are seeded into every graph schema.

  **New node types** (add to existing 6, total ~20 base types):
  ```
  Person, Organization, Location, Event, Project, Tool,
  Concept, Document, Preference, Activity, Asset, Substance,
  Metric, Symptom, Goal, Task, Emotion, Recipe, Protocol, Routine
  ```

  **New edge types** (add to existing 12, total ~25 base types):
  ```
  RELATES_TO, MENTIONS, CAUSED_BY, FOLLOWS, AUTHORED, USES,
  CONTRADICTS, SUPPORTS, SUMMARIZES, DERIVED_FROM, SUPERSEDES, CORRECTS,
  HAS_GOAL, WORKS_ON, WORKS_FOR, LOCATED_AT, PART_OF,
  EXPERIENCED, CONSUMES, PERFORMS, OWNS, RECOMMENDS, IMPROVES
  ```

  Use `ON CONFLICT (name) DO NOTHING` so this is safe to re-run on existing schemas.
  Add descriptions for all types -- the ontology agent uses descriptions to decide reuse.

### 2. Create per-domain seed type templates

- File: `src/neocortex/domains/ontology_seeds.py` (NEW)
- Details: Define domain-specific type recommendations as Python data. These are NOT
  mandatory -- they're injected into the ontology agent's context as "recommended types
  for this domain" (see Stage 3).

  Structure:
  ```python
  @dataclass
  class DomainOntologySeed:
      node_types: dict[str, str]   # {name: description}
      edge_types: dict[str, str]   # {name: description}

  DOMAIN_SEEDS: dict[str, DomainOntologySeed] = {
      "user_profile": DomainOntologySeed(
          node_types={
              "Person": "Human individual",
              "Location": "Physical place or address",
              "HealthState": "Physical or mental health condition or state",
              "Routine": "Recurring behavioral pattern or habit",
              "Dream": "Dream or sleep experience",
              "Substance": "Supplement, medication, or consumed substance",
              "Symptom": "Health symptom or condition",
              "Emotion": "Emotional or psychological state",
              "Trip": "Planned or completed journey",
              "FoodItem": "Dish, meal, or food product",
              "Ingredient": "Food substance used in cooking",
              "MediaWork": "Book, movie, podcast, or creative work",
              "Vehicle": "Car, bicycle, or transportation asset",
              "Protocol": "Structured health or behavioral intervention",
              ...
          },
          edge_types={
              "HAS_STATUS": "Current physical/mental state",
              "EXPERIENCES_SYMPTOM": "Health condition observed",
              "CONSUMES": "Substance or food consumed",
              "PERFORMS_ROUTINE": "Recurring habit or practice",
              "HAS_PARTNER": "Romantic partner relationship",
              "HAS_PET": "Pet ownership",
              "VISITS_LOCATION": "Visited a place",
              "WORKS_FOR": "Employment relationship",
              "WORKS_ON": "Active project engagement",
              "HAD_DREAM": "Dream experienced",
              ...
          },
      ),
      "technical_knowledge": DomainOntologySeed(...),
      "work_context": DomainOntologySeed(...),
      "domain_knowledge": DomainOntologySeed(...),
  }
  ```

  Each domain should have 15-25 node types and 20-30 edge types that are specific
  to the kind of knowledge that domain receives.

  See `resources/domain_seeds.md` for complete type lists per domain.

### 3. Wire domain seeds into the extraction pipeline

- File: `src/neocortex/extraction/pipeline.py`
- Details: In `run_extraction()`, after loading existing types (line ~104), also load
  the domain seed for the current domain (if `domain_hint` is set, parse the domain
  slug from it). Pass the seed types to the ontology agent deps as a new field
  `recommended_types`.

  This does NOT create the types in the database -- it only provides them as guidance
  to the ontology agent. The agent decides which to propose; the pipeline persists
  what the agent proposes.

### 4. Seed the expanded base types into existing graph schemas

- File: `migrations/graph/003_seed_ontology.sql`
- Details: Since this migration uses `ON CONFLICT DO NOTHING`, it will naturally
  add missing base types to existing schemas the next time `MigrationRunner` runs.
  No separate migration file needed -- just ensure the migration runner re-applies
  this file (it's idempotent).

  Alternatively, create `migrations/graph/NNN_expanded_seed.sql` if the migration
  runner only applies new migrations.
