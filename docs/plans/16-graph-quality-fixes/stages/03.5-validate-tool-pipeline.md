# Stage 3.5: E2E Validation — Tool-Equipped Librarian Pipeline

**Goal**: Validate the complete tool-driven extraction pipeline end-to-end, including content updates, dedup, edge stability, and partial failure recovery.

**Dependencies**: Stage 3 (mutation tools + pipeline redesign)

**Priority**: P0 — gates Stage 4

---

## Why This Validation Exists

Stages 2-3 are the core architectural change. The librarian now makes multiple
tool calls per episode instead of producing a static payload. This validation
confirms the new pipeline:
1. Produces correct graph state (no regressions from the old pipeline)
2. Handles entity updates and dedup via tools (not blind UPSERT)
3. Maintains edge type stability across re-extractions
4. Recovers cleanly from partial failures

---

## Steps

### 3.5.1 Run the full E2E test suite

```bash
./scripts/run_e2e.sh scripts/e2e_extraction_pipeline_test.py
./scripts/run_e2e.sh scripts/e2e_cognitive_recall_test.py
./scripts/run_e2e.sh scripts/e2e_content_update_test.py   # from Stage 1.5
```

All three must pass. The extraction pipeline test is the critical one — it
exercises the full ingest → extract → recall path.

### 3.5.2 Run the Plan 15 scenario replay

Create `scripts/e2e_plan15_scenarios_test.py` that replays the 14 scenarios
from Plan 15's empirical validation (Stage 6). This is the acceptance gate
for the plan's success criteria: ≥11/14 (79%) acceptable.

The scenarios test:
1. **Content updates**: entity description reflects latest information
2. **Entity dedup**: same entity mentioned across episodes → 1 node
3. **Edge type stability**: same relationship re-extracted → same edge type
4. **Node type consistency**: entity type stays stable across episodes
5. **Contradiction handling**: new info contradicts old → node updated
6. **Edge weight bounds**: no weight creep beyond ceiling

Each scenario:
1. Ingests 1-2 episodes with known content
2. Queries the graph via diagnostic SQL (from `resources/queries.md`)
3. Asserts specific graph state
4. Records pass/fail per scenario

```bash
./scripts/run_e2e.sh scripts/e2e_plan15_scenarios_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-3.5-scenarios.log
```

### 3.5.3 Verify tool call metrics

After the E2E tests, check the audit log for tool call patterns:

```bash
# Count tool calls per episode
grep "librarian_tool_call" log/agent_actions.log | \
  jq -r '.tool' | sort | uniq -c | sort -rn
```

Verify:
- Average tool calls per episode is reasonable (<30)
- `find_node_by_name` is called before `create_or_update_node` (workflow compliance)
- No episodes hit the `max_tool_calls=50` limit

### 3.5.4 Verify partial failure recovery

Simulate a partial failure by:
1. Ingesting an episode
2. Before extraction completes, kill the extraction worker
3. Re-trigger extraction for the same episode
4. Verify `cleanup_partial_curation` ran (check logs)
5. Verify final graph state is correct (no duplicate nodes from partial run)

### 3.5.5 Persist results

```bash
# Capture all results
./scripts/run_e2e.sh scripts/e2e_extraction_pipeline_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-3.5-extraction.log
./scripts/run_e2e.sh scripts/e2e_plan15_scenarios_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-3.5-scenarios.log
```

## Pass/Fail Criteria

**PASS**: All E2E tests pass AND Plan 15 scenarios ≥11/14. Proceed to Stage 4.

**FAIL with regressions**: If `e2e_extraction_pipeline_test.py` fails, the pipeline
redesign broke existing functionality. Do not proceed — fix regressions first.

**FAIL with low scenario score**: If Plan 15 scenarios <11/14, identify which
scenarios fail and whether they need librarian prompt tuning or tool logic fixes.
Document failures in the Issues section of `index.md`.

**FAIL with high tool calls**: If average >30 per episode, consider adding a
batched `find_nodes_by_names(names: list[str])` tool to reduce round-trips.
