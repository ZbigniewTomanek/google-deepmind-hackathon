# Stage 5.5: E2E Validation — Weight Management

**Goal**: Verify that edge weight changes produce stable scoring behavior end-to-end, without weight creep or scoring regressions.

**Dependencies**: Stage 5 (weight management fix)

**Priority**: P2

---

## Why This Validation Exists

Stage 5 changes three interconnected weight mechanisms: reinforcement formula,
micro-decay, and stale-edge decay window. These interact with the hybrid recall
scoring system. Unit tests verify each mechanism in isolation; this validation
confirms the scoring system produces fair, bounded results in practice.

### 5.5.2 Run a weight-stability stress test

Create `scripts/e2e_weight_stability_test.py` that:

1. Ingests 3 seed episodes with overlapping entities (shared subgraph).
2. Performs 20 sequential `recall` queries targeting the shared subgraph.
3. After each recall, records edge weights via diagnostic SQL.
4. Asserts:
   - No edge weight exceeds 1.5 (new ceiling)
   - Reinforced edges show diminishing increments (not linear)
   - Non-traversed edges in the schema have decayed at least once
   - Weight distribution has bounded variance (no single edge dominates)

```bash
./scripts/run_e2e.sh scripts/e2e_weight_stability_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-5.5-weights.log
```

### 5.5.3 Run the full E2E suite as regression check

```bash
./scripts/run_e2e.sh scripts/e2e_extraction_pipeline_test.py
./scripts/run_e2e.sh scripts/e2e_hybrid_recall_test.py
```

Weight changes affect recall scoring, which could change result ordering
in the hybrid recall test.

### 5.5.4 Check weight distribution via diagnostic SQL

```bash
psql -h localhost -U neocortex -d neocortex -f - <<'SQL'
SET search_path TO ncx_dev_user__personal;
-- Q5: Weight distribution
SELECT
  width_bucket(weight, 0, 2, 20) as bucket,
  count(*) as cnt,
  round(min(weight)::numeric, 3) as min_w,
  round(max(weight)::numeric, 3) as max_w,
  round(avg(weight)::numeric, 3) as avg_w
FROM edge
GROUP BY bucket ORDER BY bucket;

-- Q6: Top-weighted edges (should all be ≤1.5)
SELECT e.id, ns.name as source, nt.name as target,
       et.name as edge_type, round(e.weight::numeric, 3) as weight
FROM edge e
JOIN node ns ON e.source_id = ns.id
JOIN node nt ON e.target_id = nt.id
JOIN edge_type et ON e.type_id = et.id
ORDER BY e.weight DESC LIMIT 10;
SQL
```

## Pass/Fail Criteria

**PASS**: All E2E tests pass, weight bounds respected, Plan 15 ≥11/14.
The plan is complete. Update `index.md` to mark all stages DONE.

**FAIL with weight creep**: If edges still exceed 1.5, the diminishing returns
formula isn't aggressive enough. Reduce `edge_reinforcement_delta` or increase
the denominator scaling factor.

**FAIL with scoring regression**: If `e2e_hybrid_recall_test.py` fails, the
weight changes affected result ordering. Check whether the test expectations
need updating (lower weights → different activation bonuses) or whether the
decay is too aggressive.
