# Stage 1.5: E2E Validation — Content Updates

**Goal**: Verify that node content updates work end-to-end through the full extraction pipeline, not just at the unit test level.

**Dependencies**: Stage 1 (content update fix)

**Priority**: P0 — gates Stage 2

---

## Why This Validation Exists

Stage 1 fixes content updates at three layers (prompt, pipeline fallback, SQL).
Unit tests verify each layer in isolation, but the real risk is in the interaction:
does the librarian actually produce updated descriptions when processing a second
episode about the same entity?

---

## Steps

### 1.5.1 Run existing E2E extraction pipeline test

```bash
./scripts/run_e2e.sh scripts/e2e_extraction_pipeline_test.py
```

This test ingests seed texts, triggers extraction, and verifies nodes appear.
Confirm it still passes after Stage 1 changes.

**Expected**: All assertions pass. Record output.

### 1.5.2 Run a content-update-specific E2E scenario

Create a targeted test script `scripts/e2e_content_update_test.py` that:

1. Ingests text: "Alice works on the billing team as a senior engineer."
2. Waits for extraction to complete (poll episode consolidation status).
3. Queries the graph via MCP `recall` for "Alice" — verify content mentions "billing".
4. Ingests updated text: "Alice transferred to the auth team as tech lead."
5. Waits for extraction to complete.
6. Queries the graph via MCP `recall` for "Alice" — verify content mentions "auth team"
   and does NOT still say "billing team" as the primary role.
7. Verifies only 1 node named "Alice" exists (no duplicate).

```bash
./scripts/run_e2e.sh scripts/e2e_content_update_test.py
```

### 1.5.3 Persist results

Save the test output to `docs/plans/16-graph-quality-fixes/results/stage-1.5.log`:

```bash
./scripts/run_e2e.sh scripts/e2e_content_update_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-1.5.log
```

---

## Verification

- [ ] `e2e_extraction_pipeline_test.py` passes (no regressions)
- [ ] Content-update E2E scenario: Alice node content reflects "auth team" after second ingestion
- [ ] No duplicate Alice nodes created
- [ ] Results log saved to `results/stage-1.5.log`

---

## Pass/Fail Criteria

**PASS**: All 3 checks above succeed. Proceed to Stage 2.

**FAIL**: If content is stale after re-extraction, the Stage 1 fix is incomplete.
Debug by checking:
1. Did the librarian return a description? (check extraction logs)
2. Did the pipeline fallback trigger? (check `_persist_payload` logs)
3. Did `upsert_node` receive non-NULL content? (check adapter debug logs)
