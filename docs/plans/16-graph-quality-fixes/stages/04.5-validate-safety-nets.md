# Stage 4.5: E2E Validation — Adapter Safety Nets

**Goal**: Verify that adapter-level dedup catches LLM mistakes without breaking legitimate homonym handling, and confirm no regressions from Stage 4.

**Dependencies**: Stage 4 (adapter safety nets)

**Priority**: P1

---

## Why This Validation Exists

Stage 4 changes the core `upsert_node` and `upsert_edge` behavior at the adapter
layer. This is a defense-in-depth change — if the heuristic is too aggressive it
will incorrectly merge distinct entities; if too conservative it adds no value.
The semantic guard (`_types_are_merge_safe`) needs real-world validation.

---

## Steps

### 4.5.1 Run the full E2E test suite

```bash
./scripts/run_e2e.sh scripts/e2e_extraction_pipeline_test.py
./scripts/run_e2e.sh scripts/e2e_cognitive_recall_test.py
./scripts/run_e2e.sh scripts/e2e_content_update_test.py
```

All must pass. Stage 4's adapter changes affect every node/edge write path,
so regressions are likely if the two-phase lookup has bugs.

### 4.5.2 Re-run Plan 15 scenarios

```bash
./scripts/run_e2e.sh scripts/e2e_plan15_scenarios_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-4.5-scenarios.log
```

The scenario score should be ≥ the Stage 3.5 score (safety nets should only
improve things, not regress). If it's lower, the adapter is incorrectly
merging or splitting entities.

### 4.5.3 Check drift/homonym monitoring logs

```bash
# Check for type drift catches
grep "node_type_drift_caught" log/agent_actions.log | \
  jq '{name, existing_type, requested_type, action}'

# Check for homonym detections
grep "node_homonym_detected" log/agent_actions.log | \
  jq '{name, existing_type, requested_type, action}'

# Check for edge drift catches
grep "edge_type_drift_caught" log/agent_actions.log | \
  jq '{source_id, target_id, old_type_id, new_type_id}'
```

Review each logged event:
- **Drift catches**: Were they correct merges? (entity was the same, type drifted)
- **Homonym detections**: Were they correct separations? (entities were different)
- **False positives**: Any merges that should have been separations?
- **False negatives**: Any duplicates that should have been caught?

### 4.5.4 Run diagnostic SQL queries

Use the queries from `resources/queries.md` to check graph health:

```bash
# Q3: Duplicate node detection
psql -h localhost -U neocortex -d neocortex -f - <<'SQL'
-- Set search_path to the test agent's schema
SET search_path TO ncx_dev_user__personal;
-- Q3 from resources/queries.md
SELECT lower(name) as lname, count(*) as cnt
FROM node WHERE NOT forgotten
GROUP BY lname HAVING count(*) > 1
ORDER BY cnt DESC;
SQL
```

### 4.5.5 Persist results

```bash
./scripts/run_e2e.sh scripts/e2e_plan15_scenarios_test.py 2>&1 | tee \
  docs/plans/16-graph-quality-fixes/results/stage-4.5-scenarios.log
```

## Verification

- [ ] All E2E tests pass (no regressions from adapter changes)
- [ ] Plan 15 scenario score ≥ Stage 3.5 score
- [ ] Drift catches are all correct merges (manual review)
- [ ] No false positive merges of legitimate homonyms
- [ ] Diagnostic SQL shows no unexpected duplicate nodes
- [ ] Results logs saved to `results/`

---

## Pass/Fail Criteria

**PASS**: No regressions, scenario score stable or improved, no false merges.

**FAIL**: If scenario score drops, check whether `_types_are_merge_safe` is
too aggressive. Add failing type pairs to `_HOMONYM_TYPE_GROUPS` and re-run.
